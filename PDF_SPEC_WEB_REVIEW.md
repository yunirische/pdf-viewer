# PDF Spec Web UX Review

Цель этой ветки - дать дизайнеру текущий веб-интерфейс `pdf_spec` для UX-разбора.

## Что это за продукт

Пользователь загружает PDF с проектной документацией, выбирает нужные страницы, дает инструкции агенту и получает расчет по оборудованию: состав шкафа, комплектующие и сметную стоимость.

Основной пользователь - не технический специалист. Интерфейс должен объяснять порядок работы сам, без знания терминов API, OpenClaw, BOM, artifacts или внутренней логики пайплайна.

## Что сейчас путает пользователя

1. Неочевиден порядок работы.
   Пользователь не понимает, что делать первым: создать проект, загрузить PDF, выбрать страницы, извлечь текст, нажать AI, получить BOM, получить смету.

2. Термин `BOM` непонятен.
   Для пользователя лучше использовать человекочитаемые названия: `Состав оборудования`, `Комплектующие`, `Предварительная спецификация`.

3. Кнопка AI-агента выглядит как черный ящик.
   После нажатия кнопка крутится, но непонятно, идет ли обработка, зависла ли система, есть ли лимит агента, ждать ли дальше.

4. Ошибки backend/API сейчас не объясняются человеческим языком.
   Пример реальной ошибки:

   ```text
   Failed to load resource: the server responded with a status of 504
   /api/v1/projects/.../extract
   ```

   Пользователь должен видеть понятное сообщение: что именно не получилось, можно ли повторить, связано ли это с лимитом агента или долгой обработкой PDF.

5. Выбор страниц нужен как важный сценарий, а не вторичная настройка.
   Это экономит токены и время. Пользователь должен понимать, что можно указать, например: страницы 47-71, если там линейная схема и спецификация.

6. Поле инструкций агенту нужно оформить как часть основного процесса.
   Это не просто заметка. Это постановка задачи для анализа.

7. Окно просмотра PDF сейчас не регулируется.
   Пользователю неудобно сверять документ, страницы, инструкции и результат. Нужен resize/split-view или другой способ управлять размером PDF-просмотра без потери рабочего контекста.

8. Без агента PDF разбирается плохо.
   Нужно продумать отдельный сценарий: что система может сделать локальным/простым извлечением, где качество ограничено, когда нужен агент, и как объяснить пользователю разницу без технических деталей.

## Пример реальной инструкции агенту

```text
В файле есть линейные схемы начиная со страницы 47.
Спецификация начинается со страницы 71.

Стр. 47: пунктиром выделена линейная схема ВП1 5ВРУ.
Стр. 71: первая строка - ВРУ общежития, вводная панель 5ВРУ,
ВП1 2000х600х600 мм (ВхШхГ).

Задача: по линейной схеме подобрать комплектующие и рассчитать
сметную стоимость готового шкафа в сборе:
ВРУ общежития, вводная панель 5ВРУ, ВП1 2000х600х600 мм (ВхШхГ).

Производитель оборудования: Систем Электрик.
Шкаф (оболочка): ДКС.

Сметная стоимость оборудования - интернет цена / прайсовая / розничная.
Стоимость сборки для сметы: x2 от стоимости оборудования.
```

## Что желательно получить от дизайнера

- Понятный пошаговый сценарий для первого запуска.
- Нормальные названия этапов вместо технических терминов.
- Состояния загрузки: идет обработка PDF, ожидаем агента, агент недоступен, превышен лимит, можно повторить.
- Человеческие тексты ошибок.
- Логичное место для выбора страниц.
- Логичное место для инструкций агенту.
- Удобное регулируемое окно просмотра PDF.
- Понятный fallback-сценарий без агента: что доступно, что будет хуже, когда лучше дождаться агента.
- Визуальное разделение: источник PDF, задача, прогресс обработки, результат.

## Где лежит веб-копия

Копия приложения находится в:

```bash
pdf-spec-web/
```

Запуск локально:

```bash
cd pdf-spec-web
npm install
npm run dev
```

По умолчанию веб ожидает API по `/api/v1`.
Для локального override можно задать:

```bash
VITE_API_BASE_URL=http://localhost:8000/api/v1
```
